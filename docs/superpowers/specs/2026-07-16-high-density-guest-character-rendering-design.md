# 하객 캐릭터 고밀도 렌더링 설계

## 배경

현재 12개 하객 프리셋은 방향별 보행 3컷을 포함한 `96×144` 고밀도 원안을 보유한다. 그러나 게임 월드에서는 생성 단계에서 각 프레임을 최근접 보간으로 `48×72`까지 축소한 별도 시트를 사용한다. 이 과정에서 눈·코·입, 헤어 윤곽, 의상 주름과 액세서리가 크게 소실되고 흰색 다중 드롭 섀도가 인물을 스티커처럼 보이게 만든다. 반면 새 맵은 큰 원본 이미지를 논리 좌표에 맞춰 표시하므로 더 세밀하고 부드러운 픽셀 질감을 유지한다.

## 목표

- 게임 내 하객 표시 크기는 기존 `48×72`로 유지한다.
- 12명의 얼굴, 헤어, 의상과 색상 정체성을 변경하지 않는다.
- `96×144` 원안을 직접 사용해 고밀도 디테일을 보존한다.
- 상하좌우 3컷, 총 144개 보행 프레임의 크기와 3등신 비율을 유지한다.
- 밝은 실내, 어두운 예식홀, 복잡한 연회장 배경에서 모두 캐릭터를 식별할 수 있게 한다.
- 이동 좌표, 충돌 범위, 포털 접근, 카메라와 실시간 동기화 계약은 변경하지 않는다.

## 비목표

- 캐릭터를 `60×90` 이상으로 확대하지 않는다.
- 12명의 얼굴이나 의상을 새로운 화풍으로 전면 재생성하지 않는다.
- 신랑·신부 NPC의 디자인이나 렌더링 파이프라인은 변경하지 않는다.
- 기존 미추적 `guest-walk-ratio-*` 원안 디렉터리를 수정하거나 정리하지 않는다.
- 캐릭터 선택 UI의 레이아웃이나 프리셋 수를 변경하지 않는다.

## 선택한 방식

### 고밀도 원안 직접 렌더링

월드 표시에서도 `characters/generated/guests/<preset>__walk.png`와 `__idle.png`를 사용한다. 프레임 좌표 계산 기준은 원안 크기인 `96×144`, 표시 크기는 기존 `48×72`로 둔다. 브라우저가 한 프레임을 절반 크기로 합성하므로 게임상의 물리적 크기와 위치는 그대로 유지되고 고해상도 원안의 색·윤곽 정보가 더 많이 남는다.

기존 `guests/world/*` 축소 시트는 이번 변경의 런타임 경로에서는 사용하지 않는다. 생성·감사 호환성과 빠른 롤백을 위해 당장은 제거하지 않는다.

### 렌더링과 외곽선

월드 하객 레이어에 한해 `image-rendering: auto`를 사용한다. 원안 자체의 픽셀 질감은 유지하면서 축소 시 계단 현상과 얼굴 뭉개짐을 줄인다. 캐릭터 전체를 둘러싼 불투명한 흰색 네 방향 외곽선은 제거하고 다음 두 단계로 교체한다.

1. 배경에서 실루엣을 분리하는 약한 웜톤 림
2. 발밑과 오른쪽 아래에 집중된 짧은 반투명 그림자

외곽 효과는 얼굴과 의상 내부 디테일을 덮지 않아야 하며 밝은 바닥과 어두운 배경에서 모두 식별 가능해야 한다.

### 144프레임 선별 감사

원안은 기존 디자인을 유지하되 런타임 렌더링 변경 후 모든 프레임을 다시 확인한다. 자동 검사는 다음 계약을 확인한다.

- 12명 × 4방향 × 3컷 = 144컷 존재
- 각 원안 프레임 `96×144`, 게임 표시 `48×72`
- 투명 배경과 유효한 캐릭터 픽셀 존재
- 각 프레임의 불투명 바운딩 박스 상단이 `y=5~6`, 하단이 `y=132`, 높이가 `127~128px` 범위에 있음
- 월드의 가로·세로 표시 배율이 모두 정확히 `0.5`여서 비등방 변형이 없음
- 3등신 기준선 감사에서 머리·헤어 덩어리가 첫 기준선 `y≈48`, 몸과 의상이 나머지 두 구간을 차지함
- 얼굴 영역이 과도하게 납작하거나 길어지는 비등방 변형이 없음

접촉 시트에서는 각 캐릭터별 12컷과 전체 144컷을 실제 표시 크기 및 확대 보기로 함께 렌더링한다. 문제가 없는 프레임은 변경하지 않는다. 결함이 확인된 경우에만 기존 원안을 복사한 별도 후보 파일에서 국소 보정하고, 비교 감사 통과 후 프로덕션 시트에 반영한다.

## 데이터 흐름

1. `guest-character-presets.json`이 기존 원안과 표시 크기를 제공한다.
2. `generate-character-assets.mjs`가 `96×144` 원안 시트를 공개 자산 경로로 복사한다.
3. `resolveCharacterLayers`가 월드에서도 고밀도 공개 자산 URL과 원안 프레임 크기를 반환한다.
4. `CharacterSprite`가 기존 방향·보행 프레임 좌표를 계산하고 CSS로 `48×72`에 표시한다.
5. 게임 이동·충돌·카메라 코드는 기존 캐릭터 중심 좌표를 그대로 사용한다.

## 오류 처리와 롤백

- 고밀도 자산 로드 실패 시 기존 `failedUrls` 처리로 깨진 레이어를 숨기고 개발 환경 콘솔에 URL을 기록한다.
- 자동 감사나 브라우저 검수에서 가독성 저하가 발견되면 `resolveCharacterLayers`의 월드 URL만 기존 `guests/world/*`로 되돌릴 수 있다.
- 선택적 프레임 보정은 후보와 전후 비교 이미지를 남겨 원안 전체를 되돌리지 않고 해당 프레임만 복구할 수 있게 한다.

## 테스트

### 단위·자산 테스트

- 월드 모드가 고밀도 `96×144` 시트를 선택하는지 검증한다.
- 월드 표시 크기가 계속 `48×72`인지 검증한다.
- 방향별 프레임 좌표와 idle blink 좌표가 원안 시트에 맞는지 검증한다.
- `pnpm characters:audit`, `pnpm characters:test`, 전체 `pnpm test`, `pnpm typecheck`, `pnpm build`를 통과시킨다.

### 시각 검수

- 12명 각각의 상하좌우 3컷 접촉 시트를 만든다.
- 기존 48×72 축소본과 새 고밀도 렌더링을 같은 배율로 비교한다.
- 모바일 `390×844`와 데스크톱 `1440×1000`에서 우리 집, 예식홀, 연회장을 확인한다.
- 정지·보행·포털 이동·원격 하객 표시에서 프레임 흔들림, 크기 변화, 흰 테두리 과다, 배경 매몰이 없는지 확인한다.
- 이미지 요청 실패와 브라우저 콘솔 오류가 없어야 한다.

## 완료 조건

- 12명 전원이 기존과 동일한 게임 크기와 디자인을 유지한다.
- 144개 보행 프레임이 고밀도 경로로 정상 표시된다.
- 얼굴과 의상 디테일이 기존 월드 축소본보다 명확하다.
- 3등신 비율, 방향별 키, 발 기준선과 보행감이 일관된다.
- 이동·충돌·포털·미니맵·실시간 하객 동작에 회귀가 없다.
- 모바일과 데스크톱 실제 맵 검수에서 승인 가능한 결과를 얻는다.
