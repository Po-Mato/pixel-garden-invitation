# 참석 답변 실사용화 설계

## 목표

기존의 일회성 참석 답변 폼을 실제 예식 인원 집계에 사용할 수 있는 흐름으로 확장한다. 하객은 청첩장 안에서 답변을 제출하고 같은 기기에서 다시 확인·수정할 수 있어야 한다. 신랑·신부는 비밀번호로 보호된 관리 화면에서 현황을 확인하고 CSV로 저장할 수 있어야 한다.

## 확정 예식 정책

- 참석 답변 권장 마감: 2027년 4월 24일
- 마감 이후에도 제출과 수정은 허용한다.
- 개인정보 자동 삭제: 2027년 5월 31일 종료 시점 이후 첫 예약 작업
- 수집 목적: 예식 참석, 식사 및 동행 인원 확인
- 관리자 접근: Cloudflare Worker Secret으로 보호되는 비밀번호 로그인
- 하객 수정: 제출한 브라우저에 저장된 개별 수정 키 사용

## 범위

### 포함

- 신랑측·신부측 구분
- 이름과 연락처
- 참석·불참·미정 상태
- 조건부 참석 인원 및 식사 여부
- 전달사항과 개인정보 수집 동의
- 신규 제출 후 요약 확인
- 같은 기기의 기존 응답 조회와 수정
- 비공개 관리자 로그인과 단기 세션
- RSVP 요약, 검색, 필터, 응답 삭제
- 한글 Excel 호환 CSV 저장
- 기존 D1 데이터 보존 마이그레이션
- 예약 작업을 통한 RSVP 개인정보 자동 삭제
- 모바일·데스크톱·Worker API 검증

### 제외

- SMS 또는 이메일 본인 인증
- 다른 기기에서 이름·연락처로 응답 조회
- 하객이 직접 응답을 삭제하는 기능
- 관리자에 의한 응답 내용 수정
- 동행자별 이름과 개별 식사 여부
- 자가용·대중교통 이용 조사
- 외부 설문 또는 스프레드시트 동기화
- RSVP 데이터의 무기한 보관

## 하객 화면

기존 초대장 메뉴와 월드의 `참석 답변` 장소가 공통 RSVP 하단 시트를 연다. 두 진입점은 같은 상태와 API를 사용하며 시트 조작이 캐릭터 이동으로 전달되지 않아야 한다.

### 신규 답변

표시 순서는 다음과 같다.

1. `2027년 4월 24일까지 알려주세요` 마감 안내
2. `신랑측 / 신부측` 세그먼트
3. 이름
4. 연락처
5. `참석 / 불참 / 미정` 세그먼트
6. 참석 상태에 따른 인원·식사 항목
7. 전달사항
8. 개인정보 수집·보관 동의
9. 제출 버튼

필드 계약은 다음과 같다.

- 대상 구분: `groom` 또는 `bride`, 필수
- 이름: 앞뒤 공백을 제거한 1~30자, 필수
- 연락처: 구분 문자를 제거한 숫자 기준 8~15자리, 필수
- 참석 상태: `yes`, `no`, `unsure` 중 하나, 필수
- 본인 포함 인원: 1~10의 정수
- 식사 여부: `yes`, `no`, `unsure`, `not_applicable` 중 하나
- 전달사항: 앞뒤 공백을 제거한 0~160자
- 개인정보 동의: 동의 시각과 동의 문구 버전을 함께 저장

참석 상태별 동작은 다음과 같다.

- `참석`: 본인 포함 총인원과 `식사 예정 / 식사 안 함 / 미정`을 모두 입력한다.
- `불참`: 인원과 식사 항목을 숨기고 인원은 `0`, 식사는 `not_applicable`로 저장한다.
- `미정`: 예상 인원만 입력하고 식사는 `unsure`로 저장한다.

식사 상태는 응답에 포함된 전원에게 동일하게 적용한다. 관리자 집계에서 `식사 예정 인원`은 참석 응답 중 식사 상태가 `yes`인 응답의 총인원 합계다.

### 제출 완료와 수정

신규 제출이 성공하면 입력 폼 대신 다음 답변 요약을 표시한다.

- 대상 구분과 이름
- 연락처
- 참석 상태
- 적용되는 경우 총인원과 식사 여부
- 전달사항
- 최종 제출 시각
- `답변 수정` 버튼

Worker는 신규 제출 때 응답 ID와 수정 키를 반환한다. 클라이언트는 초대장 ID별로 이 두 값만 `localStorage`에 저장한다. 이후 RSVP 시트를 열 때 저장값이 있으면 본인 응답을 조회해 요약부터 표시한다.

수정 화면은 기존 값을 채운 같은 폼을 사용한다. 수정 성공 후 요약과 최종 수정 시각을 갱신한다. 같은 기기에서 브라우저 저장소를 지우면 기존 응답을 복구할 수 없으며 새 응답은 제출할 수 있다. 연락처만으로 기존 응답을 합치거나 노출하지 않는다.

### 마감과 개인정보 안내

2027년 4월 24일까지는 일반 마감 안내를 표시한다. 이후에는 `마감일이 지났지만 답변을 보내실 수 있습니다`로 바꾸며 제출과 수정은 계속 허용한다.

동의 문구는 수집 항목, 예식 인원 확인 목적, 2027년 5월 31일까지의 보관 기간과 자동 삭제 정책을 명시한다. 동의하지 않으면 제출할 수 없다.

## 관리자 화면

GitHub Pages의 직접 진입 404를 피하기 위해 운영 주소의 쿼리 기반 진입점 `?admin=rsvp`를 사용한다. 일반 초대장 메뉴에는 이 진입점을 표시하지 않는다. 주소는 편의를 위한 진입점일 뿐 보안 수단이 아니며 모든 데이터 API는 관리자 인증을 요구한다.

### 로그인

- 비밀번호 입력 하나와 로그인 버튼을 제공한다.
- 비밀번호는 저장소, Pages 번들, D1에 원문으로 저장하지 않는다.
- 배포 전에 생성한 PBKDF2 비밀번호 해시를 Worker Secret으로 설정한다.
- 로그인 성공 시 Worker가 초대장 ID와 만료 시각을 포함한 1시간 유효 서명 토큰을 반환한다.
- 토큰은 `sessionStorage`에만 저장하며 탭 종료 또는 만료 후 다시 로그인한다.
- 반복 로그인 실패는 클라이언트 주소의 keyed hash를 기준으로 제한한다. 원본 IP는 D1에 저장하지 않는다.

### 현황과 목록

관리 화면 상단에는 다음 값을 표시한다.

- 전체 응답 수
- 참석 확정 응답 수
- 참석 확정 총인원
- 식사 예정 총인원
- 불참 응답 수
- 미정 응답 수와 예상 인원
- 자동 삭제 예정일

목록은 최신 수정 순으로 표시하고 이름, 대상 구분, 참석 상태, 인원, 식사, 연락처, 전달사항, 생성·수정 시각을 포함한다. 데스크톱은 표, 좁은 모바일은 항목별 행 레이아웃을 사용한다. 전체 화면을 중첩 카드로 구성하지 않는다.

다음 제어를 제공한다.

- 이름 또는 연락처 검색
- 신랑측·신부측 필터
- 참석·불참·미정 필터
- 식사 여부 필터
- 필터 초기화
- 확인 절차를 거친 단일 응답 삭제
- 현재 필터가 아닌 전체 응답 CSV 저장

관리자가 응답 내용을 직접 바꾸는 기능은 제공하지 않는다. 중복이나 잘못된 응답은 삭제하고 하객이 자신의 기기에서 다시 제출하도록 한다.

### CSV

CSV는 UTF-8 BOM과 CRLF 줄바꿈을 사용해 한글 Excel에서 바로 열 수 있게 한다. 파일에는 내부 수정 키 해시, 관리자 토큰, 클라이언트 식별값을 포함하지 않는다. 셀 값이 `=`, `+`, `-`, `@`로 시작하면 앞에 작은따옴표를 붙여 수식 실행을 막는다.

열 순서는 다음과 같다.

```text
대상, 이름, 연락처, 참석 상태, 총인원, 식사 여부, 전달사항, 최초 제출, 최종 수정
```

## 데이터 모델

새 D1 마이그레이션은 기존 `rsvps`를 호환 가능한 새 테이블로 옮긴 뒤 교체한다. 기존 CHECK 제약이 인원 `0`을 허용하지 않으므로 단순 열 추가만 사용하지 않는다.

새 RSVP 행은 다음 의미를 가진다.

- `id`
- `invitation_id`
- `side`: `groom`, `bride`, `legacy`
- `guest_name`
- `phone`: 정규화된 연락처, 기존 행은 nullable
- `attendance`: `yes`, `no`, `unsure`
- `party_size`: 0~10
- `meal_status`: `yes`, `no`, `unsure`, `not_applicable`
- `note`
- `consent_version`: 신규 행 필수, 기존 행은 nullable
- `consented_at`: 신규 행 필수, 기존 행은 nullable
- `edit_token_hash`: 신규 행 필수, 기존 행은 nullable
- `revision`: 낙관적 동시성 제어용 정수
- `created_at`
- `updated_at`

기존 응답은 이름, 참석 상태, 인원, 메모와 생성 시각을 보존한다. 대상은 `legacy`, 식사는 `unsure`, 연락처·동의·수정 키는 `NULL`로 옮긴다. 기존 응답은 관리자 화면에서 `이전 형식`으로 확인할 수 있지만 하객 수정 대상은 아니다.

`invitations`에는 RSVP 권장 마감과 개인정보 삭제 시점을 UTC ISO 값으로 저장한다. 운영 초대장은 2027년 4월 24일 종료와 2027년 5월 31일 종료에 대응하는 값을 가진다. 화면 표시와 Worker 예약 삭제가 같은 값을 사용한다.

관리자 로그인 제한을 위해 짧게 보관되는 시도 집계 테이블을 추가한다. 클라이언트 주소는 Worker Secret으로 HMAC 처리한 값만 저장하고 제한 창이 지난 행은 예약 작업에서 제거한다.

## API 계약

### 하객 API

- `POST /api/invitations/:invitationId/rsvps`: 신규 제출, 응답 ID와 수정 키 및 정규화된 응답 반환
- `GET /api/invitations/:invitationId/rsvps/:rsvpId`: 수정 키 검증 후 본인 응답 반환
- `PATCH /api/invitations/:invitationId/rsvps/:rsvpId`: 수정 키와 현재 revision 검증 후 수정

본인 조회와 수정은 `Authorization: Bearer <edit-token>`을 사용한다. D1에는 SHA-256 기반 수정 키 해시만 저장한다. 키는 충분한 엔트로피를 가진 Web Crypto 난수로 생성하며 응답 본문과 브라우저 저장소 외에는 원문을 남기지 않는다.

수정 요청은 마지막으로 읽은 `revision`을 포함한다. 서버 값과 다르면 `409 Conflict`를 반환하고 클라이언트는 최신 응답을 다시 불러오도록 안내한다.

### 관리자 API

- `POST /api/invitations/:invitationId/admin/session`: 비밀번호 검증 및 단기 토큰 발급
- `GET /api/invitations/:invitationId/admin/rsvps`: 요약과 필터 가능한 응답 목록 반환
- `DELETE /api/invitations/:invitationId/admin/rsvps/:rsvpId`: 확인된 단일 응답 삭제

관리자 데이터 API는 서명된 관리자 Bearer 토큰을 요구한다. 토큰에는 초대장 ID와 만료 시각을 포함하고 다른 초대장의 데이터에는 사용할 수 없다. 응답에는 `Cache-Control: no-store`와 민감 응답에 맞는 보안 헤더를 적용한다.

API CORS는 운영 Pages origin과 명시적인 로컬 개발 origin만 허용한다. 허용 메서드는 필요한 `GET`, `POST`, `PATCH`, `DELETE`, `OPTIONS`로 제한하고 `content-type`, `authorization` 헤더만 허용한다.

## 자동 삭제

Worker에 일일 예약 작업을 추가한다. 작업은 다음 순서로 실행된다.

1. 현재 시각이 `invitations.rsvp_delete_at`을 지난 초대장을 찾는다.
2. 해당 초대장의 RSVP 행을 삭제한다.
3. 만료된 관리자 로그인 시도 집계도 삭제한다.
4. 삭제된 건수를 구조화 로그로 남긴다. 개인정보 값은 로그에 남기지 않는다.

삭제는 동일 작업이 반복 실행돼도 안전해야 한다. 하객 조회는 이미 삭제된 응답에 `404`를 반환하고, 관리 화면은 0건 집계를 정상 상태로 표시한다.

## 오류 처리

- 제출 실패: 입력값과 동의를 유지하고 재시도할 수 있게 한다.
- 중복 버튼 입력: 처리 중 제출·수정 버튼을 비활성화한다.
- 입력 오류: 서버와 클라이언트가 같은 조건부 필드 규칙을 검증한다.
- 손상된 수정 키: `401`을 받은 클라이언트는 저장값을 제거하고 신규 폼으로 복구한다.
- 삭제된 응답: `404`를 받은 클라이언트도 저장값을 제거하고 삭제 사실을 알린다.
- 수정 충돌: `409`에서 최신 응답을 다시 불러오고 사용자가 다시 확인하게 한다.
- 전송 제한: `429`에서 잠시 후 재시도 안내와 재활성화 시점을 표시한다.
- 관리자 토큰 만료: 로그인 화면으로 돌아가되 검색·필터 UI 상태는 유지한다.
- 목록 실패: 로그인 상태를 유지하고 목록 영역에서 다시 불러오기를 제공한다.
- CSV 실패: 화면 데이터는 유지하고 저장 실패만 알린다.

## 접근성·반응형

- 참석 상태와 대상 구분은 키보드로 조작 가능한 세그먼트 또는 라디오 그룹으로 구현한다.
- 조건부 필드가 나타날 때 기존 포커스를 빼앗지 않는다.
- 필드 오류는 입력과 연결하고 제출 결과는 `aria-live`로 알린다.
- 전화번호 입력은 `type="tel"`과 적절한 모바일 입력 모드를 사용한다.
- 하단 시트는 기존 포커스 트랩, `Escape`, 닫기 후 포커스 복원을 유지한다.
- 320px·390px 세로 화면에서 버튼과 필드 문구가 잘리지 않아야 한다.
- 짧은 가로 화면에서는 시트 본문만 스크롤되고 제목과 닫기 동작에 접근할 수 있어야 한다.
- 관리자 목록은 좁은 화면에서 가로 스크롤 표를 강제하지 않고 읽기 쉬운 행 레이아웃으로 전환한다.

## 테스트

### 공통 검증

- 참석 상태별 인원·식사 규칙
- 이름, 연락처, 인원, 메모 길이 제한
- 개인정보 동의 필수와 동의 버전
- 마감 이후에도 유효한 제출 허용

### 마이그레이션

- 기존 RSVP 행의 필수 데이터 보존
- 기존 행의 `legacy` 변환
- 신규 CHECK 제약과 0명 불참 허용
- 초대장별 마감·삭제 시각 저장
- 마이그레이션 재실행 안전성

### Worker API

- 신규 제출과 수정 키 해시 저장
- 올바른 키의 본인 조회·수정
- 키 누락·오류·다른 응답 키 차단
- revision 충돌
- 조건부 payload 위조 차단
- 관리자 비밀번호 성공·실패·시도 제한
- 관리자 토큰 서명·초대장 범위·만료 검증
- 관리자 요약·검색·필터·삭제
- CORS origin과 허용 헤더
- 예약 삭제의 시각 경계와 반복 실행

### 클라이언트

- 상태별 조건부 필드 렌더링
- 개인정보 동의 전 제출 차단
- 제출 중 중복 입력 차단과 실패 시 값 유지
- 제출 성공 요약과 수정 전환
- 수정 키 저장·복원·손상 복구
- 충돌과 삭제 응답 처리
- 관리자 로그인·만료·필터·검색·삭제 확인
- CSV BOM, 열 순서, 따옴표와 수식 주입 방어
- 하단 시트 입력 중 게임 이동 차단

### 브라우저·회귀

- 320px, 390px 세로 및 짧은 가로 화면에서 하객 흐름 확인
- 관리자 데스크톱 표와 모바일 행 레이아웃 확인
- 실제 Worker에서 제출·새로고침·수정·관리자 집계·CSV·삭제 확인
- 기존 일정 저장, 오시는 길, 방명록, 맵 이동, 포털, 조이스틱 회귀 확인
- 전체 테스트, 타입 검사, 프로덕션 빌드 통과

## 배포

1. 운영 D1을 내보내 백업한다.
2. 새 D1 마이그레이션을 로컬 DB에 적용하고 테스트한다.
3. 사용자가 정한 관리자 비밀번호의 PBKDF2 해시와 별도 세션 서명 키를 Worker Secret으로 설정한다.
4. 운영 D1 마이그레이션을 적용한다.
5. 예약 작업과 새 API를 포함한 Worker를 배포한다.
6. API 신규·수정·관리자 흐름을 운영 origin에서 점검한다.
7. 클라이언트를 GitHub Pages에 배포한다.
8. 공개 URL에서 하객 제출·수정과 관리자 로그인·CSV를 점검한다.

운영 데이터가 있는 D1 마이그레이션과 Worker 배포를 Pages보다 먼저 수행해 새 클라이언트가 구형 API를 호출하는 시간을 만들지 않는다. 비밀번호 원문과 서명 키는 문서, 커밋, CI 로그에 남기지 않는다.

## 완료 기준

- 하객이 승인된 기본 항목으로 답변을 제출할 수 있다.
- 참석 상태에 따라 인원과 식사 항목이 정확히 변한다.
- 같은 브라우저에서 본인 답변만 다시 확인하고 수정할 수 있다.
- 권한 없는 요청으로 RSVP 원문을 조회할 수 없다.
- 관리자가 비밀번호 로그인 후 정확한 집계와 목록을 확인할 수 있다.
- 관리자가 중복 응답을 삭제하고 안전한 CSV를 저장할 수 있다.
- 2027년 5월 31일 이후 예약 작업이 RSVP 개인정보를 자동 삭제한다.
- 기존 RSVP 데이터와 기존 청첩장 기능이 마이그레이션 후 유지된다.
- 자동 테스트와 운영 환경 브라우저 검증을 통과한다.
