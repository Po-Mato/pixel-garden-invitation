# 실제 하객 동선 월드 설계

## 목표

- 청첩장 월드를 실제 하객이 집에서 출발해 예식과 연회까지 이동하는 흐름으로 재구성한다.
- 집, 외부, 지하철, 예식장 앞, 로비, 신부 대기실, 화장실, 예식홀을 포함해 최소 8개 이상의 서로 다른 맵을 제공한다.
- 맵마다 장소에 맞는 서로 다른 가로·세로 크기와 시각 구성을 사용한다.
- 모든 맵 전환은 포털을 실제로 밟았을 때만 발생하며 직접 맵 이동 버튼은 제공하지 않는다.
- 포털을 클릭하면 캐릭터가 타일 경로를 따라 포털까지 걸어가고, 도달한 뒤에만 다음 맵으로 전환한다.
- 예식홀처럼 큰 맵에서는 캐릭터가 맵 뷰포트 중앙에 유지되고 월드가 반대 방향으로 이동하는 추적 카메라를 사용한다.

## 하객 여정과 맵 크기

기본 여정은 다음 순서를 따른다.

1. **집** `480x600`: 현관, 거실, 초대장과 출발 포털
2. **동네 거리** `960x540`: 주택가, 횡단보도, 가로수, 지하철역 입구
3. **지하철 역사** `720x720`: 개찰구, 노선 안내, 에스컬레이터, 승강장 포털
4. **지하철 차량** `1080x480`: 긴 객차, 좌석, 창밖 이동 풍경, 하차문
5. **예식장 앞** `840x720`: 건물 전경, 드롭오프, 꽃 간판, 로비 입구
6. **예식장 로비** `960x780`: 안내 데스크, 축의대, 포토월, 방명록, 갈림길
7. **신부 대기실** `600x540`: 신부 NPC, 소파, 거울, 꽃 장식, 로비 복귀 포털
8. **예식홀** `660x1800`: 입구, 긴 버진로드, 하객 좌석, 단상, 연회장 포털
9. **화장실** `540x600`: 세면대, 거울, 칸막이, 로비 복귀 포털
10. **연회장** `1080x840`: 원형 테이블, 뷔페, 축하 메시지, 여정 마무리

집에서 예식장 로비까지는 선형 흐름이다. 로비에서는 신부 대기실, 화장실, 예식홀로 분기한다. 신부 대기실과 화장실은 로비로 돌아오며, 예식홀 끝에서 연회장으로 이동한다. 모든 목적지는 역방향 포털을 제공해 이미 지나온 장소로도 걸어서 돌아갈 수 있다.

## 장소별 시각 방향

- **집:** 따뜻한 살구빛 실내, 창문으로 들어오는 아침빛, 현관 타일과 작은 화분
- **동네 거리:** 청록 새벽 하늘, 벽돌길, 횡단보도, 편의점과 가로수
- **지하철 역사:** 민트·회색 타일, 노란 안전선, 개찰구 조명, 컬러 노선 띠
- **지하철 차량:** 크림색 객차, 청록 좌석, 손잡이, 창문을 지나는 도시 불빛
- **예식장 앞:** 아이보리 건물, 코랄 꽃 간판, 발렛 구역, 유리문과 화단
- **로비:** 진주 대리석, 금빛 안내 데스크, 꽃 포토월, 축의대와 대기 소파
- **신부 대기실:** 로즈·아이보리, 꽃벽, 큰 거울, 소파와 드레스 조명
- **예식홀:** 짙은 녹색과 금빛, 긴 아이보리 버진로드, 코랄 꽃 장식, 단상 조명
- **화장실:** 밝은 민트·라일락 테라조, 큰 거울, 금속 세면대, 명확한 출구
- **연회장:** 샴페인·청록·코랄, 원형 테이블, 뷔페 조명, 축하 가랜드

각 맵은 공통 프리즘 픽셀 디테일을 공유하되 바닥 재질, 구조물, 길의 형태, 장식 밀도를 다르게 한다. 검은 굵은 테두리는 포털과 상호작용 요소에만 사용하고 환경 장식은 주변색 외곽선을 사용한다.

## 월드 데이터 모델

`WorldZoneId`는 다음 10개 ID를 허용한다.

```text
home, neighborhood, subway-station, subway-train, venue-exterior,
lobby, bridal-room, ceremony-hall, restroom, banquet
```

`WorldZone`은 기존 `bounds`, `spawn`, `blocked`, `spots`, `npcs`, `portals`, `decorations`에 다음 정보를 추가한다.

- `theme`: 장소별 CSS 테마 식별자
- `cameraSafeBounds`: 플레이어가 이동할 수 있는 카메라 안전 영역
- `pathStyle`: 중앙 통로, 복도, 객차, 버진로드 등 장소별 길 표현
- `journeyIndex`: 비클릭 여정 표시 순서

`WorldPortal`은 다음 정보를 추가한다.

- `approach`: 캐릭터가 포털 전환 전에 도달해야 하는 타일 좌표
- `facing`: 도달 시 캐릭터가 바라볼 방향
- `spawn`: 대상 맵에서 카메라 안전 영역 안에 있는 도착 좌표

포털의 시각 영역은 벽, 문, 계단, 차량 문처럼 표현할 수 있지만 실제 이동 목표는 항상 `approach` 타일이다.

## 포털 전용 이동

상단의 맵 이동 버튼은 제거한다. HUD에는 현재 장소, 여정 단계, 지나온 장소를 보여주는 비클릭 경로표만 남긴다. 기존 초대장 메뉴는 정보 모달 바로가기로만 유지하며 맵 이동 기능을 갖지 않는다.

포털 클릭 흐름은 다음과 같다.

1. 사용자가 포털 버튼을 클릭한다.
2. 현재 플레이어 타일에서 포털의 `approach` 타일까지 이동 가능한 경로를 계산한다.
3. 경로가 있으면 `targetPortalId`와 타일 경로를 저장한다.
4. 기존 보행 프레임과 타일 이동 간격을 사용해 경로의 다음 타일로 순서대로 이동한다.
5. 각 타일 도착 후 현재 위치와 `approach`를 비교한다.
6. 정확히 도달하면 캐릭터 방향을 `facing`으로 바꾸고 짧은 문 열림 상태를 표시한다.
7. 그 시점에만 대상 맵과 `spawn`으로 전환한다.

포털 클릭만으로 즉시 `moveToZone`을 호출하지 않는다. 포털 이동 중 사용자가 조이스틱을 움직이거나 다른 맵 지점을 클릭하면 자동 경로와 `targetPortalId`를 즉시 취소한다. 다른 포털을 클릭하면 기존 경로를 버리고 새 포털 경로를 계산한다.

경로가 없으면 현재 맵을 유지하고 HUD에 `길을 찾을 수 없어요` 상태를 짧게 표시한다. 포털은 접근 타일과 연결 통로가 다른 충돌 영역과 겹치지 않도록 월드 데이터 검사에서 검증한다.

## 타일 경로 탐색

기존 `computeNextGridPosition`은 한 타일의 이동 유효성 검사와 수동 조작에 계속 사용한다. 포털 자동 보행은 검증된 그리드 A* 경로 탐색 라이브러리를 사용한다.

- 맵 `bounds`를 현재 타일 크기의 그리드로 변환한다.
- `blocked`, 상호작용 카드, 통과 불가능한 장식을 장애물 셀로 표시한다.
- 시작점과 `approach`를 그리드 좌표로 변환한다.
- 상하좌우 이동만 허용하고 대각선 이동은 금지한다.
- 결과 경로를 기존 월드 좌표 타일 목록으로 변환한다.
- 경로 계산 결과는 포털 클릭 시 한 번만 만들고 이동 중에는 재계산하지 않는다.

## 추적 카메라

`world-map`은 폰 안에서 고정된 뷰포트 역할만 하고, `world-map__stage`가 구역의 실제 논리 크기를 갖는다. 모든 맵 요소는 실제 월드 좌표로 배치한다.

카메라는 다음 값을 계산한다.

```text
cameraX = viewportCenterX - playerX * zoom
cameraY = viewportCenterY - playerY * zoom
```

스테이지에는 `translate3d(cameraX, cameraY, 0) scale(zoom)`을 적용한다. 로컬 플레이어는 스테이지 안의 월드 좌표를 유지하므로 화면에서는 중앙에 보이고 배경, 포털, 장식, NPC, 다른 하객이 함께 이동한다.

모든 플레이 가능 좌표와 포털 접근 타일은 `cameraSafeBounds` 안에 둔다. 이 영역은 뷰포트 절반 이상의 시각 여백을 맵 가장자리에 확보하므로 플레이어가 실제 이동 가능한 끝에 도달해도 카메라를 강제로 멈추지 않고 중앙을 유지한다. 집이나 신부 대기실처럼 상대적으로 작은 맵도 뷰포트보다 큰 논리 크기와 안전 여백을 갖는다.

맵 클릭 좌표는 다음 역변환을 사용한다.

```text
worldX = (screenX - viewportLeft - cameraX) / zoom
worldY = (screenY - viewportTop - cameraY) / zoom
```

화면 회전이나 뷰포트 크기 변경 시 카메라를 즉시 다시 계산한다. 카메라 이동은 타일 보행과 동일한 `steps()` 감각을 유지하며 큰 흐림이나 부드러운 관성 스크롤을 사용하지 않는다.

## 실시간 하객 동기화

- 로컬 위치는 기존처럼 월드 좌표와 `zoneId`로 전송한다.
- 포털 자동 보행 중에도 각 타일 이동을 기존 throttle로 전송한다.
- 대상 맵의 `zoneId`는 포털 접근 타일에 도달해 실제 전환한 뒤에만 전송한다.
- 다른 구역의 하객은 렌더링하지 않고 같은 구역의 하객만 현재 카메라 변환 안에서 표시한다.
- 10개 구역 ID를 공유 validation과 worker 상태에서 모두 허용한다.

## 상호작용 콘텐츠 배치

- `directions`: 집 또는 지하철 역사
- `wedding-info`: 예식장 로비
- `gallery`, `story`: 로비 포토월
- `couple`: 신부 대기실의 신부와 예식홀 단상의 신랑
- `rsvp`: 예식장 로비 축의대
- `guestbook`: 연회장 축하 테이블

기존 초대장 정보와 모달 데이터는 유지하고 장소만 실제 동선에 맞게 재배치한다.

## 접근성과 상태 표현

- 포털은 실제 `button`을 유지하고 목적지 이름을 접근성 이름으로 제공한다.
- 자동 보행 시작, 취소, 길 찾기 실패, 맵 도착을 `aria-live` 상태로 알린다.
- 키보드 사용자는 포털 버튼을 활성화해 동일한 자동 보행을 시작할 수 있다.
- 모션 감소 설정에서도 타일 위치 변화는 유지하되 장식 애니메이션과 문 반짝임은 정지한다.
- 경로표는 비클릭이며 현재 위치에 `aria-current="location"`을 사용한다.

## 검증 기준

- 월드는 정확히 10개 구역 ID를 공유 프로토콜과 데이터에 가진다.
- 모든 구역 크기가 동일하지 않고 가로형, 세로형, 정사각형에 가까운 맵이 함께 존재한다.
- 집에서 연회장까지 포털 그래프가 연결되고 신부 대기실·화장실 분기가 로비로 복귀한다.
- 모든 포털의 `approach`와 `spawn`이 타일 그리드, 카메라 안전 영역, 비충돌 영역 안에 있다.
- 포털 클릭 직후에는 구역이 바뀌지 않고 자동 보행이 시작된다.
- 포털 접근 타일에 도달한 경우에만 구역과 실시간 `zoneId`가 바뀐다.
- 조이스틱 또는 다른 맵 클릭이 포털 자동 보행을 취소한다.
- 장애물이 있는 맵에서도 경로 탐색이 포털까지 유효한 상하좌우 타일 경로를 만든다.
- 예식홀 `660x1800`에서 버진로드 전체를 이동하는 동안 플레이어 중심과 카메라 역변환이 유지된다.
- `320x568`, `390x844`, `430x932`에서 HUD, 카메라 뷰포트, 조작부가 겹치지 않는다.
- 10개 맵을 직접 이동하며 포털 도달, 역방향 이동, 분기, 긴 맵 카메라를 브라우저로 확인한다.
- 전체 테스트, 타입 검사, 프로덕션 빌드가 통과해야 한다.
